01 · introduction

Autonomous cloud governance & FinOps automation.

ZopNight is an autonomous cloud governance platform that enforces cost discipline through policy-driven remediation, enabling engineering teams to move fast while maintaining financial control.

Built by ZopDev, ZopNight moves beyond traditional FinOps dashboards. Where legacy tools surface insights after the damage is done, ZopNight prevents waste at its source through automated scheduling, dependency-aware resource orchestration, and 337+ audit rules across AWS, GCP, Azure, and Kubernetes.

What ZopNight does

  • Enforces cost discipline automatically through cron-based start/stop schedules that power down non-production resources during idle hours, verified to reduce cloud spend by up to 60%.
  • Detects and remediates waste with idle resource detection, right-sizing recommendations, orphan cleanup, and discount opportunity identification across all three major cloud providers.
  • Orchestrates infrastructure safely with dependency-aware execution ordering, production-safe overrides, and full audit trails for every state transition.
  • Scales infrastructure proactively with VM autoscaling policies, event readiness pre-scaling, and cost anomaly detection that catches budget deviations before they compound.
  • Provisions multi-cloud infrastructure end-to-end via ZopDay, with single-wizard deployment of Kubernetes clusters, datastores, networking, and add-ons across AWS, GCP, and Azure.

Architecture overview

ZopNight follows a discover-schedule-execute pattern. All API requests pass through a gateway that handles JWT validation and org context injection, routing to five backend services:

  • Config: Schedules, resource groups, overrides, cloud accounts, autoscaler policies, event readiness definitions.
  • Discoverer: Multi-cloud resource discovery, metrics collection, activity log sync.
  • Executor: Start/stop execution, state history, autoscaler and event readiness action fulfillment.
  • Aggregator: Cost reports, trends, budgets, showback, billing sync, anomaly detection.
  • Recommender: 337+ audit rules, optimization recommendations, auto-tagging predictions.
Production-safe by design

ZopNight performs start and stop operations only. It never deletes resources. Production environments receive automatic protection flags, and every action is logged with full attribution for audit readiness.

Who this document is for

Engineering leaders evaluating ZopNight, platform teams onboarding their cloud accounts, and FinOps practitioners building cost governance programs. Covers every capability in depth, from five-minute quick-start to advanced autoscaling policies and API integration patterns.

02 · getting started

Five minutes to first value.

ZopNight is designed for a five-minute path to first value. Connect a cloud account, discover resources, attach a schedule, and start enforcing cost discipline immediately.

Prerequisites

  • A ZopNight organization (created during signup via Google, GitHub, or SAML SSO).
  • Cloud provider credentials with appropriate permissions (read-only for discovery, read-write for start/stop enforcement).
  • At least one non-production environment (dev, staging, QA) with running resources.

Quick-start workflow

  1. Connect a cloud account. Add AWS, GCP, or Azure credentials. ZopNight supports static keys, assume-role (AWS), service account JSON (GCP), service principal or Workload Identity Federation (Azure).
  2. Discover resources. ZopNight scans all regions and surfaces inventory by provider, account, region, and resource type. Discovery runs on a six-hour cycle and supports on-demand refresh.
  3. Create a schedule. Define a cron-based schedule with timezone support. Common patterns: weekday business hours, night shutdown, weekend shutdown.
  4. Attach resources. Assign individual resources or entire resource groups. Dependency-aware ordering ensures databases start before application servers.
  5. Verify and enforce. Monitor execution via the scheduler events log, notification channels (Slack, Teams, Google Chat, email), and the cost dashboard for real-time savings tracking.
AWS one-click setup

For AWS accounts, ZopNight provides a CloudFormation template that provisions the required IAM role with least-privilege permissions in a single click. No manual policy configuration required.

Permission model

Each cloud account connection specifies a permission level that governs what ZopNight can do:

LevelCapabilities
read_onlyResource discovery, cost tracking, recommendations, scheduling (without enforcement).
read_writeFull access: discovery, scheduling enforcement (start/stop), autoscaler policy application, provisioning.

Permission visibility is built in. During discovery, ZopNight classifies and records any permission errors per provider type and region, surfacing them in a dedicated permissions drawer on each cloud account card with Granted, Denied, and Unknown status badges.

03 · scheduling & automation

The scheduling engine.

The scheduling engine is the foundation of ZopNight's cost governance. It evaluates cron expressions every minute, queuing start and stop intents against cloud provider APIs with dependency-aware execution ordering and idempotent operation guarantees.

Schedule configuration

Each schedule is defined by a name, description, IANA timezone, and one or more cron entries. Standard five-field cron expressions control when resources start and stop. A single schedule can carry multiple cron entries to handle complex patterns like weekday business hours combined with extended hours on specific days.

PatternCron expressionUse case
Weekday business hours0 9 * * 1-5 / 0 18 * * 1-5Dev and staging active during work hours only
Night shutdown0 20 * * * / 0 7 * * 1-5Power down all non-production after 8 PM
Weekend shutdown0 18 * * 5 / 0 9 * * 1Full weekend shutdown for QA and staging
Extended sprint hours0 7 * * 1-5 / 0 22 * * 1-5Longer uptime during sprint cycles

Resource groups and dependency ordering

Resource groups enable coordinated lifecycle management for resources that depend on each other. Exclusive membership is enforced: one resource belongs to one group, one scheduling source per resource.

Within a group, execution ordering is defined through tiers and per-tier sequencing. ZopNight supports both automatic ordering (storage first, compute second, applications third) and custom user-defined sequences with configurable start/stop delays per resource.

  • Tiered execution: Tier 1 resources complete before Tier 2 begins. Within each tier, resources execute in defined sequence order.
  • Dependency example: A typical web application group starts the RDS database (Tier 1), then the EC2 application servers (Tier 2), then the load balancer health checks register (Tier 3). Stop order reverses automatically.
  • Bulk operations: Start or stop all resources in a group with a single action. A worker pool of four parallel executors handles concurrent operations while respecting tier ordering.

Override system

Overrides provide temporary exceptions to active schedules for production incidents, maintenance windows, demos, or ad-hoc testing.

TypeBehaviorUse case
force_onKeep resource running, ignore all stop schedulesProduction incident requiring staging environment for debugging
force_offKeep resource stopped, ignore all start schedulesMaintenance window, security patching

Overrides accept ISO 8601 expiry timestamps with timezone support. A maximum override duration policy prevents indefinite overrides from accumulating. Every override is logged with creator email, timestamp, reason, and expiry for full audit trail compliance. Overrides can be applied at both individual resource and group level.

Actions and execution

Every state transition is recorded as an action with full attribution. Actions can be triggered by schedules (cron evaluation), manual operations (API or UI), overrides, or autoscaler policies.

  • Manual actions: POST /actions triggers immediate start or stop on individual resources.
  • Bulk actions: POST /actions/bulk operates across entire resource groups with dependency-aware ordering.
  • Idempotent operations: Starting an already-running resource or stopping an already-stopped resource succeeds without side effects.
  • State tracking: Every transition (ON to OFF, OFF to ON) is recorded with timestamp, trigger source, and actor identity. The resource_state_history table provides the authoritative record for cost calculation and compliance auditing.
04 · intelligence & optimization

337+ rules. Per-resource savings.

ZopNight's intelligence layer pre-computes recommendations via cron, analyzes 14 days of cloud metrics, and surfaces actionable remediation with per-resource savings estimates. The system evaluates 337 audit rules across AWS (155), GCP (75), and Azure (107).

Severity levels

Each recommendation includes a title, description, current monthly cost, optimized cost, potential savings, evidence JSON (the cloud monitoring values that triggered the rule), and an action type with payload for one-click remediation where supported.

Auto-tagging

ZopNight's auto-tagger predicts environment classifications (prod, staging, dev) and operational flags (noStop) using deterministic, rule-based logic across naming patterns, existing tags, and instance configuration. No machine learning — predictions are transparent and reproducible.

  • Confidence scoring: 1–100. High (80–100) is bulk-acceptable; medium (50–79) warrants individual review; low (below 50) is hidden by default.
  • Non-destructive workflow: All predictions require explicit accept or reject before tags are merged. Accepted tags can optionally sync back to the cloud provider.
  • Filtering: By provider, resource type, confidence band, acceptance status.

VM autoscaling

ZopNight's autoscaler manages capacity across AWS Auto Scaling Groups, AWS ECS services, GCP Managed Instance Groups, and Azure Virtual Machine Scale Sets with a progressive adoption model.

Operating modes

ModeBehaviorUse case
MonitorCollect metrics, publish what-if recommendationsEvaluate potential savings before committing
RecommendGenerate recommendations with one-click applyControlled optimization with manual approval
AutopilotApply changes automatically within configured boundsHands-off capacity management within min/max/cooldown guardrails

Policy configuration includes minimum and maximum capacity, target CPU percentage, cooldown seconds between scaling actions, and cloud account scoping. Smart defaults are derived from 14-day historical analysis using Welford-based online statistics (P95 CPU, peak instance counts) to suggest optimal values. Schedule triggers support business-hours, peak-hours, and weekend-scale-down presets.

Adopt-or-replace for existing policies

ZopNight never silently overwrites existing cloud-side scaling configurations. When the create wizard detects pre-existing policies on a target, the user explicitly chooses between adopting the existing configuration (zero-mutation observation) or replacing it with a ZopNight-managed policy. The full pre-existing configuration is captured for byte-accurate restoration on removal.

Event readiness

Event Readiness enables proactive infrastructure pre-scaling for planned traffic spikes such as product launches, sales events, or seasonal peaks.

  • Capacity engine: Configure a traffic multiplier (e.g., 3× baseline) or expected request volume. ZopNight calculates per-target scaled min/max/desired capacity with admin override.
  • Cloud-native scheduling: Pre-scale actions use native cloud scheduled actions (AWS PutScheduledUpdateGroupAction, Azure FixedDate profiles, GCP scalingSchedules) for reliable execution without polling.
  • Lifecycle management: draft → scheduling → scheduled → scaling_up → active → scaling_down → completed. Cancellation is safe at any stage.
  • Automatic rollback: Original min/max/desired values are snapshotted at scheduling time and restored on completion or cancellation.
  • Supported targets: AWS ASG, AWS ECS services, GCP MIG, Azure VMSS. Databases can be attached as monitor-only targets with sizing recommendations.
05 · cost management & visibility

Dual-source cost tracking.

ZopNight provides dual-source cost tracking, anomaly detection, team budgets, and cost attribution for predictable, audit-ready financial governance.

Dual-source cost tracking

  • Rack rate (calculated): Computed from cloud provider pricing APIs (AWS Pricing, Azure Retail Prices). Always available for every discovered resource, providing a baseline cost view.
  • Billing cost (actual): Sourced from cloud billing APIs (AWS Cost Explorer, Azure Cost Management, GCP BigQuery). Available when billing sync is configured. Azure uses amortized cost to correctly distribute reservation and savings-plan purchases across the term.

When all cloud accounts in an organization have successful billing sync, ZopNight automatically switches all aggregate queries to billing cost. Any account missing billing data triggers a fallback to rack rate across the entire organization for consistency. The active cost source is clearly labeled in every dashboard and report.

Cost anomaly detection

The anomaly detection engine runs daily, evaluating cost deviations across five dimensions: organization, cloud account, resource group, individual resource, and team.

SeverityDeviationResponse
Warning30–150% above rolling averageInvestigation recommended
Critical150–500% above rolling averageImmediate review required
Emergency>500% deviation or new resource typeEscalation triggered

Each anomaly includes root cause attribution, identifying instance resizes, new resource creation, reservation expirations, schedule execution failures, and unscheduled usage increases. False positive guards include a minimum four data-point requirement, a $1/day cost threshold, and standard deviation exceeding 10% of the mean. Team redistribution suppression prevents false alerts when costs shift between teams with net change below 20%. A seven-day cooldown on identical anomaly keys and a top-10 resource cap per organization limit notification noise.

Team budgets

Budgets can be set per resource, resource group, or team. Current spend is calculated from cost records using the active cost source (billing or rack rate). Budget status: green (on track), yellow (at threshold), or red (over budget) with configurable alert thresholds.

Cost attribution (showback)

  • Team attribution: Cost is split equally across owning teams for shared resources. Unattributed resources are tracked separately. Endpoints provide team list, team detail (by provider and resource type), team resources (paginated), and trends.
  • Tag attribution: Cost is allocated by cloud resource tags (AWS tags, GCP labels, Azure tags) and accepted auto-tags. Each tag key-value pair receives the full resource cost. Tag origin is tracked as "cloud" or "auto."

Cost flow visualization

The Cost Reports page includes a Sankey diagram view that renders spend as a four-column flow. Layout presets include Provider, Account, Type, and Team with configurable column ordering. Clicking any node or link filters the chart and writes a breadcrumb chip for interactive drill-down. A savings overlay highlights reclaimable spend with deep-links to the recommendations page filtered to the relevant subset.

09 · cloud account management

Multiple auth methods. Per provider.

ZopNight supports multiple authentication methods per cloud provider, each with credential rotation detection, real-time sync visibility, and billing access configuration.

AWS authentication

  • Static access keys: IAM access key and secret key pair for straightforward connectivity.
  • Assume role: Cross-account STS role assumption for organizations managing multiple AWS accounts with a central control plane.
  • Temporary credentials: Short-lived STS tokens for ephemeral or tightly controlled access patterns.
  • CloudFormation template: One-click IAM role provisioning with least-privilege permissions.

GCP authentication

  • Service account JSON: Standard service account key file for programmatic access.
  • OAuth 2.0: Three-step flow with encrypted token exchange for user-delegated access.

Azure authentication

  • Service principal: Application registration with client secret or certificate.
  • Workload identity federation: Federated OIDC authentication with no stored secrets. ZopNight provides OIDC discovery endpoints for Azure WIF configuration.

Security posture

  • AES-256-GCM encryption for all stored tokens and credentials
  • No plaintext secret exposure in any API response or log output
  • Async credential verification with status polling
  • Credential rotation detection with proactive alerts
  • Billing access gap detection for cost tracking completeness

11 · security & access control

RBAC at the gateway.

ZopNight enforces role-based access control at the API gateway level, ensuring every endpoint requires appropriate authorization. The RBAC model supports three system roles and unlimited custom roles with resource-scoped permissions.

System roles

RolePermissionsScope
AdminAll policies, all operationsAll resources (organization-wide)
EditorCRUD on resources, schedules, groups; view-only on admin functionsAll resources or scoped to specific resource IDs
ViewerRead-only across all featuresAll resources or scoped to specific resource IDs

Custom roles and resource scoping

Custom roles support per-policy resource scoping. Each policy (resource, schedule, resource-group, override, cloud-account, notification, team, role, user, organisation, assignment, state-history, report, audit-log, recommendation, dashboard) can be granted with either all-resources access or scoped to specific resource UIDs. Team-scoped access allows team members to inherit their role's policies scoped to the team's assigned resource UIDs, enabling multi-tenant access patterns within a single organization.

Authentication

  • SSO integration: Google OAuth, GitHub OAuth, and SAML/SSO for enterprise identity providers (Azure AD and others).
  • Personal access tokens: API authentication via PAT tokens (prefixed zn_pat_*) for CI/CD integration and MCP server access.
  • SAML configuration: Internal admin operations for managing customer SAML IdP configurations per email domain.

Audit trail

Every mutating API request (POST, PUT, PATCH, DELETE) is logged to per-org Redis Streams, capturing both request and response bodies. The aggregator syncs to persistent storage every minute. The audit log is accessible via API and frontend with server-side filtering by method, resource, status, and user.

Read-only by design

The MCP server enforces read-only access. All write operations (schedule creation, resource modifications, override application) are restricted to the web UI and direct API. AI assistants can query and analyze without risk of unintended state changes.

Authentication requires an Admin or Editor role with a Personal Access Token. RBAC is enforced at the gateway level; the MCP server itself is a stateless proxy. In-memory TTL caching (2–30 minute windows) optimizes expensive aggregation queries.

15 · faq

Frequently asked questions.

Does ZopNight work with Auto Scaling Groups?

Yes. ZopNight sets desired capacity to zero during off-hours and restores it when the schedule fires. Your ASG configuration, scaling policies, and launch templates remain untouched.

Can I protect production environments?

ZopNight includes production safeguards enforced via RBAC. You can restrict which environments and resources can be scheduled, and auto-tagging predicts production vs. non-production labels to prevent accidental shutdowns.

What permissions does ZopNight require?

Stop and start capabilities only. ZopNight never requires delete permissions on any resource. You can start with read-only access for discovery and cost visibility, then upgrade to read-write when ready to enforce schedules.

Does it support Kubernetes workloads?

Yes. ZopNight supports EKS, GKE, and AKS clusters. It can scale node pools, manage Deployment and StatefulSet replicas, and suspend/resume CronJobs while respecting dependency ordering.

What if I need a resource running outside the schedule?

Create a one-time override via the dashboard, API, or notification channel. Overrides carry automatic expiry timestamps and take precedence over all active schedules until they expire or are manually removed. Every override is logged with creator, reason, and timestamp.

How is ZopNight different from a FinOps dashboard?

Dashboards surface the problem after the fact. ZopNight enforces the solution through policy-driven automation. Rather than reporting on idle resources, ZopNight automatically stops them on schedule and restarts them when needed. The cost reports then track verified savings from that automation.

Can I integrate ZopNight with CI/CD pipelines?

Yes. The full REST API (80+ endpoints) supports integration with any CI/CD system. Common patterns include triggering resource starts before test suites, stopping environments after deployment validation, and managing staging infrastructure programmatically.
