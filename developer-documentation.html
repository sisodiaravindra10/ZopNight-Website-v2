01 · introduction

API-first by design.

Every feature accessible in the ZopNight web UI is available programmatically. The REST API is built on standard HTTP semantics with JSON request and response bodies, bearer-token authentication, and idempotent operations for safe retry semantics.

What you can build

  • Cost automation: Trigger schedule overrides from incident webhooks, pause environments during deploys, restore them on rollout completion.
  • FinOps reporting: Pull cost trends, anomalies, and showback data into BI tools, internal dashboards, or Slack digests.
  • Resource lifecycle: Discover, schedule, and retire infrastructure programmatically — useful for ephemeral preview environments.
  • AI integrations: Connect Claude Code, Cursor, Codex, Windsurf, or Antigravity via the MCP server for conversational ops.

Base URL

All API endpoints share the same base. Production:

EnvironmentURL
Productionhttps://api.zopnight.com/v1
Sandboxhttps://api.sandbox.zopnight.com/v1
Production-safe defaults

Every API client gets read-only access by default. Write scopes (start, stop, override, schedule mutations) require an explicit role grant. Production resources carry a separate protection flag — mutations against them require an additional confirmation header.

02 · quickstart

First API call in five minutes.

Sign up, mint a Personal Access Token, and run your first request against the live API.

1 · Create an organization

Sign up via Google, GitHub, or SAML SSO at app.zopnight.com. The first user becomes Admin; additional members can be invited from Settings → Users & Teams.

2 · Mint a Personal Access Token

Navigate to Settings → Personal Access Tokens. PATs are prefixed zn_pat_ and are shown once on creation — copy them immediately. Scope tokens to the minimum role needed (Viewer, Editor, or Admin).

3 · Call the API

List your connected cloud accounts:

MethodPath
GET/v1/cloud-accounts

Example with curl:

curl https://api.zopnight.com/v1/cloud-accounts \
  -H "Authorization: Bearer zn_pat_xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx" \
  -H "Content-Type: application/json"

Response (truncated):

{
  "data": [
    {
      "id": "ca_01HQ7K3...",
      "provider": "aws",
      "name": "prod-us-east",
      "region": "us-east-1",
      "permission": "read_write",
      "sync_status": "ok",
      "last_synced_at": "2026-04-30T11:42:17Z"
    }
  ],
  "pagination": { "total": 12, "limit": 50, "offset": 0 }
}
03 · authentication

Bearer tokens, end-to-end encrypted.

ZopNight supports three authentication paths. All tokens are encrypted at rest with AES-256-GCM and validated at the API gateway before any request reaches a backend service.

Personal Access Tokens (PAT)

Recommended for service-to-service, CI/CD, and MCP integrations. PATs inherit the role of the user who created them and can be revoked individually.

  • Format: zn_pat_ prefix + 32 base32 characters
  • Header: Authorization: Bearer zn_pat_...
  • Lifetime: User-defined expiry (default 90 days, max 1 year)
  • Scopes: Inherits the minting user's role at request time (revocation propagates within 60 seconds)

Session JWTs

Issued by the login endpoint for browser sessions. Short-lived access token (15 min) plus refresh token (7 days). Use PATs for non-interactive clients.

curl -X POST https://api.zopnight.com/v1/auth/login \
  -H "Content-Type: application/json" \
  -d '{"email": "you@example.com", "password": "..."}'

SAML SSO

Configure your IdP (Azure AD, Okta, OneLogin, Google Workspace) under Settings → SSO. ZopNight provides SP metadata and ACS URL per email domain. After SAML assertion, users receive a session JWT.

Token rotation

Compromised PATs can be revoked immediately from Settings → Personal Access Tokens. The audit log records every token mint, use, and revoke event.

04 · api conventions

Predictable shapes everywhere.

Request format

  • Content type: application/json on all POST / PUT / PATCH bodies.
  • Date format: RFC 3339 / ISO 8601 with timezone offset, e.g. 2026-04-30T11:42:17Z.
  • IDs: Prefixed ULIDs — res_, sched_, grp_, ovr_, ca_, act_.

Pagination

List endpoints use offset-based pagination with these query parameters:

ParameterDescriptionDefault
limitMax items per page (max 200)50
offsetSkip N items0
sort_bySort field (varies per resource)created_at
sort_orderasc or descdesc

Filtering

Resource list endpoints accept multi-select filter parameters with comma-separated values. Selecting a higher-level filter (e.g. provider) cascades and narrows downstream filter options through dedicated /filter-fields endpoints.

GET /v1/resources?provider=aws,gcp&status=running&region=us-east-1,us-west-2

Idempotency

Start and stop operations are idempotent — starting a running resource or stopping a stopped one returns 200 with no side effects. For non-idempotent calls (resource group creation, schedule mutation), pass an Idempotency-Key header. Repeated requests within 24 hours return the original response.

Versioning

The API is versioned in the URL path (/v1/). Breaking changes ship under a new version. Deprecated endpoints remain available for at least 12 months with deprecation headers (Sunset, Deprecation) on responses.

13 groups. 80+ operations.

Every UI feature has a corresponding API operation. Full per-endpoint schemas, parameters, and response examples live in the OpenAPI document at /v1/openapi.json.

Schedules

MethodPathDescription
GET/v1/schedulesList all schedules
POST/v1/schedulesCreate with cron + timezone + entries
PATCH/v1/schedules/:idUpdate cron entries, timezone, or metadata
DELETE/v1/schedules/:idDelete (auto-detaches resources)
POST/v1/schedules/:id/attachAttach resources or groups
POST/v1/schedules/:id/detachDetach resources or groups

Resource groups, overrides, actions

CRUD operations on /v1/resource-groups, /v1/overrides, /v1/actions. Bulk action endpoint accepts a group ID and orchestrates dependency-aware execution: POST /v1/actions/bulk.

Reports & analytics

Cost trends, summaries, anomalies, scheduler events — all under /v1/reports/*. Most accept a granularity parameter (daily, weekly, monthly) and a date range.

Recommendations

337+ audit rules surface as recommendations: GET /v1/recommendations. Filter by category (idle, rightsizing, schedule, orphan, compliance, discount, governance), severity, or provider. Apply a recommendation with POST /v1/recommendations/:id/apply.

Notifications, users, roles, audit

Channel and subscription management under /v1/notifications/*. User and team operations under /v1/users and /v1/teams. Custom RBAC under /v1/roles. Audit log queries under /v1/audit-logs.

06 · webhooks

Push, not poll.

Subscribe to lifecycle events — resource state transitions, budget alerts, anomaly detections — and receive HTTP POSTs to your endpoint within seconds.

Configuration

Create a webhook subscription:

POST /v1/notifications/webhooks
{
  "url": "https://hooks.example.com/zopnight",
  "events": ["resource.started", "resource.failed", "budget.exceeded"],
  "scope": { "type": "organization" }
}

Event types

EventDescription
resource.startedResource transitioned ON via schedule or manual action
resource.stoppedResource transitioned OFF via schedule or manual action
resource.failedStart or stop operation failed
resource.overrideOverride applied or cleared
resource.retry_exhaustedAll retry attempts exhausted
budget.warningTeam budget approaching threshold
budget.exceededTeam budget exceeded
cost.anomaly.*Cost anomaly detected (per dimension and severity)

Signature verification

Every webhook delivery includes an X-ZopNight-Signature header containing an HMAC-SHA256 of the raw body using your subscription's signing secret. Reject any payload that fails verification.

// Go example
import "crypto/hmac"
import "crypto/sha256"
import "encoding/hex"

mac := hmac.New(sha256.New, []byte(secret))
mac.Write(body)
expected := hex.EncodeToString(mac.Sum(nil))
if !hmac.Equal([]byte(expected), []byte(headerSig)) {
    return errors.New("invalid signature")
}

Retries & idempotency

Failed deliveries (non-2xx response, timeout) retry up to 5 times with exponential backoff: 30s, 2m, 10m, 1h, 6h. Each retry carries the same X-ZopNight-Delivery-Id — use it as an idempotency key to avoid double-processing.

07 · sdks & libraries

Type-safe clients in three languages.

Go

Official Go SDK built on GoFr, ZopDev's open-source microservice framework. Includes typed request/response structs, automatic retry, and signature verification helpers for webhooks.

go get github.com/zopdev/zopnight-go

import "github.com/zopdev/zopnight-go"

client := zopnight.New(os.Getenv("ZN_PAT"))
resources, err := client.Resources.List(ctx, &zopnight.ListResourcesParams{
    Provider: []string{"aws"},
    Status:   "running",
})

Python

Async-first Python SDK using httpx. Type hints derived directly from the OpenAPI schema.

pip install zopnight

from zopnight import ZopNight

zn = ZopNight(token=os.environ["ZN_PAT"])
resources = await zn.resources.list(provider=["aws"], status="running")

TypeScript / Node

npm install @zopnight/sdk

import { ZopNight } from "@zopnight/sdk";

const zn = new ZopNight({ token: process.env.ZN_PAT });
const { data } = await zn.resources.list({ provider: ["aws"], status: "running" });
OpenAPI & codegen

The full OpenAPI 3.0 spec lives at https://api.zopnight.com/v1/openapi.json. Generate clients in any language with openapi-generator or import directly into Postman / Insomnia.

08 · cli & ci/cd

Pipeline-friendly by default.

Command-line interface

The zn CLI wraps the API for ad-hoc operations and CI scripts.

# Install
brew install zopdev/tap/zn

# Authenticate
export ZN_PAT="zn_pat_..."

# List staging resources
zn resources list --provider aws --tag env=staging

# Trigger a manual override
zn override create --resource res_01HQ... --action force_on --expires "2h"

# Pause all schedules in a group
zn schedules pause --group grp_01HQ...

GitHub Actions

- name: Pause staging during deploy
  uses: zopdev/zopnight-action@v1
  with:
    token: ${{ secrets.ZN_PAT }}
    action: override
    resource-group: grp_staging
    expires: 30m

Common CI/CD patterns

  • Pre-deploy: Force-on staging environments before integration tests run, force-off after they finish.
  • Preview environments: Provision via ZopDay API on PR open, attach a 24h auto-stop schedule, deprovision on PR close.
  • Cost gating: Block merges when projected cost exceeds team budget (use /v1/reports/projection).
  • Compliance: Reject deploys when a resource has a Critical recommendation open.
09 · mcp server

43 read-only tools. Five assistants.

The Model Context Protocol (MCP) server exposes ZopNight as a tool surface for AI coding assistants. Streamable HTTP transport, JSON-RPC 2.0, read-only by design.

Configure your assistant

Add the MCP server to your assistant's config file:

{
  "mcpServers": {
    "zopnight": {
      "url": "https://mcp.zopnight.com/v1",
      "headers": {
        "Authorization": "Bearer zn_pat_..."
      }
    }
  }
}

Supported assistants

AssistantConfig location
Claude Code~/.config/claude/mcp.json
CursorSettings → Features → MCP
Codex~/.codex/mcp.json
WindsurfSettings → MCP servers
Antigravity~/.config/antigravity/mcp.json

Tool surface

  • Resources & cloud accounts: list, filter, status, sync state.
  • Costs & billing: trends, breakdowns, anomalies, showback.
  • Schedules & execution: active schedules, recent events, upcoming actions.
  • Recommendations: open recommendations with savings estimates.
  • Audit log: recent mutations with full attribution.
Read-only by design

The MCP server enforces read-only access. Schedule mutations, override application, and resource modifications are restricted to the web UI and direct REST API. AI assistants can analyze freely without risk of unintended state changes.

10 · error handling

Predictable error shapes.

Status codes

CodeMeaningAction
400Validation errorInspect errors[] for field-level details
401Missing or invalid tokenRefresh token or re-mint PAT
403Authorized but lacks scopeCheck role assignments
404Resource not foundVerify ID and org context
409Conflict (e.g. duplicate name)Adjust input or reconcile state
422Business-rule violationSee error code for resolution path
429Rate limit exceededHonor Retry-After header
500Server errorRetry with backoff; report if persistent

Error payload format

{
  "error": {
    "code": "schedule.cron_invalid",
    "message": "Cron expression must have 5 fields",
    "request_id": "req_01HQ7K3..."
  },
  "errors": [
    { "field": "cron_entries[0].expression", "code": "format", "message": "expected 5 fields, got 4" }
  ]
}

Always log the request_id when reporting issues — support uses it to trace through every service hop.

11 · rate limits

Generous defaults, burst-friendly.

Rate limits are scoped per organization and per token. Limits scale with paid tier; sandbox is fixed.

TierPer-minutePer-hourBurst
Sandbox601,00030
Starter1205,00060
Pro60030,000200
EnterpriseNegotiatedNegotiatedNegotiated

Headers

Every response includes:

  • X-RateLimit-Limit — current per-minute ceiling
  • X-RateLimit-Remaining — calls left in the window
  • X-RateLimit-Reset — UNIX timestamp when window resets
  • Retry-After (on 429 only) — seconds to wait before retrying

Bulk operations

Most heavy operations — bulk start/stop, recommendation refresh, cost report regeneration — run asynchronously and don't count against rate limits past the initial enqueue. Use the returned job ID with GET /v1/jobs/:id to track progress.

12 · changelog

API changelog.

Breaking changes ship under a new version (/v2/ on next major). Additive changes, new endpoints, and new optional fields are released continuously and announced here.

2026-04

  • Added /v1/event-readiness/* for proactive pre-scaling plans.
  • Anomaly detection severity tier emergency now triggers above 500% deviation.
  • MCP server tools expanded to 43 (added budget, audit-log, and notification tools).

2026-03

  • OpenAPI 3.0 spec is now self-hosted at /v1/openapi.json.
  • Personal Access Tokens support custom expiry up to 1 year.
  • Webhook signature header renamed from X-Signature to X-ZopNight-Signature (old header still emitted until 2026-09).

2026-02

  • Cost endpoints now return amortized cost on Azure when billing sync is configured.
  • Resource discovery added 28 Azure types (Container Apps, Function Apps, Synapse).
  • Auto-tagging confidence bands tuned: high (80–100), medium (50–79), low (<50).
