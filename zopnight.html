capabilities

Nine ways ZopNight finds the money you're losing.

Scheduling is the headline. The rest is what keeps you saving after week one.

Schedules & resource groups

5-field cron with per-schedule IANA timezones. Resource groups run in tiered execution order (DBs tier 1, app servers tier 2). Scheduler evaluates every minute.

  • Per-schedule timezones
  • Tiered dependency ordering
  • Business-hours / weekend presets

Recommendations engine

337+ audit rules across AWS (155), GCP (75), Azure (107). Seven categories: idle, rightsizing, schedule, orphan, compliance, discount, governance. Five severity tiers from Info to Critical.

  • 14-day metric lookback
  • Per-resource savings estimates
  • open → applied / dismissed flow

VM autoscaling

Three modes: Monitor (what-if), Recommend (one-click apply), or Autopilot (auto within bounds). Targets ASGs, ECS, MIGs, VMSS, Databricks. Smart defaults from p95 CPU history.

  • Min / max / target CPU policies
  • Cooldown + source-aware tolerance
  • Schedule-driven scale presets

Cost anomaly detection

Catches deviations across account, service, resource type, and team/tag dimensions. Four severity tiers (Info / Warning / Critical / Emergency) with root-cause attribution and 7-day dedup.

  • Resize, restart, lapse attribution
  • 20%-200%+ deviation thresholds
  • Suppressed-dot UI dedup

Dashboards & reports

Four preset dashboards (Executive, Engineering, FinOps, All) plus 10 widgets, 12 report types, and Sankey / treemap visualizations. CSV, PNG, and shareable URL exports.

  • Cost trend, breakdown, showback

  • 50 dashboards · 50 widgets each

Auto-tagging

Deterministic rule-based prediction (no ML) for prod / staging / dev plus a noStop operational flag. Confidence 1-100 with reasoning. Bulk-accept high, individually review medium, hide low.

  • Confidence-tiered review flow

RBAC, SSO & audit

Role-based permissions with read-only access preview. SSO via Google, GitHub, SAML. AES-256-GCM token encryption. Production auto-protected. Full audit trail of who, what, when, why.

faq

Things teams ask before they sign.

Will ZopNight ever touch production? +

Only if you explicitly tell it to. Production is gated by an explicit allow-list and a separate "production-aware" flag. Most customers never enable production scheduling — the savings are in dev / staging / sandbox where 70% of waste lives.

What permissions does ZopNight need? +

Read-only on day one — we inventory and recommend without changing anything. Once you review the plan, grant scoped write permissions per resource type (e.g. ec2:StopInstances, rds:StopDBInstance). Nothing wider than the action requires. Kubernetes uses a namespaced ServiceAccount, never cluster-admin.

What happens during an incident? +

Hit the override switch in the UI or call POST /v1/override. Every scheduled action pauses for the window you set. ZopNight also auto-pauses if it detects a deploy through your CI/CD or a PagerDuty incident in scope — we wire those in during onboarding.

How is this different from native AWS / GCP / Azure scheduling? +

Native schedulers are single-cloud, single-service, and have no dependency awareness. ZopNight is multi-cloud, dependency-ordered (DBs before apps), and adds the bits the providers don't — right-sizing, orphan cleanup, per-team showback, override APIs, and a unified audit trail.

How long until we see savings? +

Idle and orphan resources surface in the first 60-second scan — that's day-one, $50–300/mo per account in cleanup. Scheduling savings show up the first night the rule fires (typically 24 hours after onboarding). Right-sizing is week-two, after we've collected enough metrics to be confident.

Is there a free tier? +

Yes. Up to 25 monitored resources, all features, no time limit. We don't gate cost-saving features behind a paywall — if you have a small footprint, we'd rather you save money than pay us.
