Closed-loop IAM remediation: how DERA closes the 14-day security hole.

RM
Riya Mittal Engineer · ZopDev
· Apr 27, 2026 · 8 min read

The median time from detection to remediation in cloud security is 14 days. That number isn't a constraint of the cloud — it's a constraint of the workflow that wraps it. Detect, ticket, triage, assign, fix, verify, close. Every hop costs hours. The gap between "we know about it" and "the door is shut" is where attackers live.

For the last year we've been running an internal experiment: what if the workflow had no humans in the middle? Not "human approves automation" — fully closed-loop. Detect, evaluate, remediate, audit. We call it DERA, and over six months it took our internal IAM-drift remediation time from 14 days to under 90 seconds.

This is a walk-through of how DERA works, what it auto-fixes, what it explicitly leaves alone, and how the audit trail satisfies SOC 2 without anyone signing off.

01The detect-then-ticket model has a 14-day security hole.

Most cloud security tooling stops at detection. AWS Config flags the over-permissive role; the SIEM ingests the event; a Jira ticket is opened. That ticket then waits — for a security engineer to triage it, an owner to be identified, a fix to be authored, a code review to land, a deploy window to open. Median time-in-queue at most companies we've measured is 11 days. The remaining 3 days are spent verifying the fix.

From an auditor's standpoint, this is fine — there's a record, an owner, a closure. From an attacker's standpoint, this is generous. A misconfigured role exposed to the internet for 14 days is 14 days of working time.

The premise of detect-then-ticket

is that automated remediation is risky enough to gate behind a human. The premise of DERA is that humans are risky enough to gate around — provided the automation knows what it's allowed to touch and what it isn't.

02The DERA loop applies closed-loop automation to security.

DERA is four stages, each of which is independently scoped and audited:

  1. Detect — AWS Config, GCP Security Command Center, Azure Defender, or a custom rule fires. The finding lands in our queue with a stable ID and full context.
  2. Evaluate — A disposition rule decides what to do. Three outcomes: auto-remediate, auto-remediate-with-grace, or escalate-to-human. The disposition is deterministic and version-controlled.
  3. Remediate — If auto, an SSM Automation document (or equivalent) runs the fix. If grace, the fix runs after a window (typically 5 minutes) during which the resource owner can object via Slack.
  4. Audit — Every action — detection, evaluation, decision, remediation outcome — is written to an immutable log with full input/output capture. The audit record is what an auditor reads, and it's what regenerates the state if anything fails.
DERA loop · per-finding latency

The same shape applies to cost remediation (which is how we got here). If you've used ZopNight's autopilot, you've used DERA — same engine, different rules.

03IAM over-permissions are scoped down automatically.

The first IAM finding we wired into DERA was AWS IAM Access Analyzer's "external access" detection. When a role is granted access from outside the account boundary — say a wildcard principal in a trust policy — Access Analyzer fires within minutes. The finding includes the analyzer ID, the resource ARN, and the principal that triggered it.

Our evaluation pipeline then asks three questions in order:

  • Is this resource on the break-glass exemption list? (Hard-coded set of role names that we explicitly never auto-remediate.)
  • Has this resource been used by the suspicious principal in the last 7 days? (Pulled from CloudTrail.)
  • Was this trust policy changed by an automation (e.g. Terraform) within the last hour? (If so, defer — the change is likely intentional and will reconcile.)

If all three are negative, the role's trust policy is updated to remove the offending principal. The fix runs in ~12 seconds via SSM. The role's owner gets a Slack message with the diff, the disposition, and a one-click rollback URL.

04Security group drift gets closed in under 90 seconds.

The second finding type is security group drift — when a security group rule drifts from the IaC-declared state. AWS Config has a managed rule for this (configservice:vpc-sg-restricted-ssh), and the response is straightforward: revert the rule to match the source-of-truth.

Our automation document does this in three steps:

# 1. fetch the IaC-declared state from terraform-state bucket
aws s3 cp s3://tf-state/sg-prod-web.json .

# 2. compute the diff against current
aws ec2 describe-security-groups --group-ids sg-0a1b2c | \
  jq '.SecurityGroups[0].IpPermissions' > current.json

diff sg-prod-web.json current.json > drift.diff

# 3. revoke unauthorized ingress, re-authorize declared
aws ec2 revoke-security-group-ingress --cli-input-json file://drift.diff
aws ec2 authorize-security-group-ingress --cli-input-json file://sg-prod-web.json

The whole thing runs in 60–80 seconds depending on the number of rule changes. The audit record captures the input diff, the output diff, the SSM execution ID, and the owner who was notified.

05What you must never auto-remediate.

The single most important part of DERA is the exemption list. The list is shorter than you'd expect, but everything on it is on it for a reason.

  • Break-glass roles. The roles your incident-response team uses to bypass normal access controls. If automation revokes a break-glass role mid-incident, you've made the incident worse. Tag these with security:no-auto-remediate=true and DERA will skip them.
  • Active sessions. Don't revoke a role while it has an active CloudTrail trail. If you must, pause for 60 seconds to let the operation finish.
  • The root account. Never. Not even with grace. Page a human.
  • Database admin grants. The risk of breaking a write-path mid-transaction outweighs the risk of leaving an over-broad grant for an extra hour.

The exemption list is committed to the same repo as the disposition rules. Adding to it requires two reviewers, same as any change. There's no Slack-shortcut.

06The audit trail satisfies SOC 2 without a human signing off.

This is the part that surprised our auditors. SOC 2's Common Criteria 7.2 requires that "security events are detected and responded to in a timely manner." The control isn't "a human reviews each event" — it's "a documented process responds in a timely, traceable way."

DERA produces a record per event with seven fields: finding_id, detection_source, disposition, disposition_reason, remediation_action, remediation_outcome, operator. operator is the automation's identity, not a human. Our auditor's question was: "what if the disposition rule is wrong?" The answer is: the rules are version-controlled, change-reviewed, and the audit trail captures which version of the rules was used at the time of evaluation.

"The control isn't a human reviews each event — it's a documented process responds in a timely, traceable way."

The result: faster mean-time-to-remediation, smaller window of exposure, and an audit trail that's actually better than the human-in-the-loop version, because nothing falls between the cracks of "I'll get to that ticket tomorrow."

07Where to start.

If you want to try this on your own infrastructure, start in observation mode. Wire up the detection, the disposition rules, and the audit log. Have the disposition output what it would have done, and then compare against what your humans did over the same period. If the dispositions agree with the humans 95%+ of the time on a given finding type, it's safe to enable auto-remediation for that type.

For the IAM and security-group rules above, we found agreement was >99% within two weeks. Cost remediation took longer (8–12 weeks of observation) because the variance in what's "OK" is much higher.

If you want to skip the build and just have it work, ZopNight ships DERA pre-baked for the cost surface, and we'll set up security-side DERA for design partners on request.
